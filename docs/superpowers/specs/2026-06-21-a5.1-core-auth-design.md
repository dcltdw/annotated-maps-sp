# A5.1 ‚Äî Core Authentication ‚Äî Design

- **Status:** approved (brainstorming) ‚Äî 2026-06-21
- **Goal:** Replace the unauthenticated `preview_as` identity stand-in with **real email/password accounts + bearer-token sessions**, so the app uses a session-authenticated identity for visibility + writes. The `preview_as` persona switcher survives **only** as an anonymous public-demo feature (gated by `SANDBOX_MODE`).
- **Relationship:** the first sub-project of **A5** (the "real auth" slice the `preview_as` seam was always a placeholder for ‚Äî `FIXME(A5)` in `core/visibility/resolve.py`). Built on the merged feature core (read + own-CRUD + appends + section-visibility + region notes + A4 sandbox/moderation). Independent of the (still-pending) A4 deploy.
- **Key decisions (locked in brainstorming):** (a) auth is **additive** ‚Äî it coexists with the demo persona switcher, not replacing it; (b) **email + password**, open signup **plus** seeded persona logins for the demo; (c) **self-implemented on Django's vetted primitives** (the `django.contrib.auth.hashers` password hashers + our own session tokens) ‚Äî no turnkey auth package, and we **extend the existing custom `User`** rather than adopt `django.contrib.auth`'s User; (d) **opaque bearer token** transport (`Authorization: Bearer`) ‚Äî **no CSRF surface**, decoupled from the A4 anonymous cookie-session.

## 1. Data model

Extend the existing `core.models.User` (referenced as the author/identity FK everywhere ‚Äî additive fields only, no rework):
- `email = models.EmailField(unique=True, null=True, blank=True)` ‚Äî nullable+unique so pre-existing/test users without an email don't collide (Postgres permits multiple NULLs under a unique constraint). Real accounts always have one.
- `password = models.CharField(max_length=128, blank=True, default="")` ‚Äî a Django password-hasher string (set via `make_password`); blank means "no password / cannot log in" (e.g. a legacy persona before seeding adds creds).

New **`AuthSession`** model (in `core/models.py`, extends `BaseModel`):
- `user = models.ForeignKey(User, on_delete=models.CASCADE, related_name="auth_sessions")`
- `token_hash = models.CharField(max_length=64, unique=True)` ‚Äî **SHA-256 hex of the raw token**. We store only the hash, so a DB leak cannot yield usable tokens. A fast hash is correct here (the token is 256-bit random ‚Äî unlike a password, it needs no slow KDF).
- `expires_at = models.DateTimeField()`
- `created_ip = models.GenericIPAddressField(null=True, blank=True)` + `user_agent = models.CharField(max_length=300, blank=True, default="")` (optional, for "your sessions" later).

One migration. No `django.contrib.auth` app install ‚Äî `make_password`/`check_password` work as a library (they read `settings.PASSWORD_HASHERS`, which has a Django default; if the standalone call needs the app, add `django.contrib.auth` to `INSTALLED_APPS` minimally ‚Äî implementation detail, verified in the plan).

## 2. Auth API ‚Äî `core/auth_api.py` router, mounted at `/api/v1/auth`

- `POST /auth/signup` `{email, password, display_name}` ‚Üí validate (email format + not already taken; password length ‚â• 8) ‚Üí `User.objects.create(email=‚Ä¶, password=make_password(pw), display_name=‚Ä¶)` ‚Üí mint an `AuthSession` ‚Üí `201 {token, user:{id,email,display_name}}` (auto-login). Duplicate email ‚Üí `409`; weak/invalid input ‚Üí `422`.
- `POST /auth/login` `{email, password}` ‚Üí look up by email, `check_password` ‚Üí on success mint an `AuthSession` ‚Üí `200 {token, user}`. On any failure (unknown email OR wrong password) ‚Üí **generic `401 "Invalid email or password."`** (no user enumeration). A user with a blank password can never log in.
- `POST /auth/logout` (bearer required) ‚Üí delete the presenting `AuthSession` ‚Üí `204`. Unauthenticated ‚Üí `401`.
- `GET /auth/me` ‚Üí the bearer user `{id,email,display_name,reputation}`, or `401`. The SPA calls this on load to restore/clear its session.

**Token:** `secrets.token_urlsafe(32)` (‚âà256-bit), returned raw exactly once; persisted as its SHA-256. **Expiry:** `now + 14 days` (fixed). Logout deletes the session row; an expired/absent/unknown token ‚Üí unauthenticated.

## 3. Identity resolution ‚Äî the seam

A helper `authed_user(request) -> User | None`: parse `Authorization: Bearer <token>` ‚Üí SHA-256 ‚Üí fetch a non-expired `AuthSession` ‚Üí its user (else `None`).

A single resolver replaces the scattered `preview_as` identity logic (~18 spots in `maps/api.py`):

```
resolve_identity(request, preview_as) -> UUID | None:
    user = authed_user(request)
    if user is not None:
        return user.id                      # authenticated identity ALWAYS wins
    if settings.SANDBOX_MODE and preview_as is not None:
        return preview_as                   # demo-only persona preview (anonymous)
    return None                             # guest
```

- **Reads:** `resolve_viewer(resolve_identity(request, preview_as), tenant)` ‚Äî the visibility engine is unchanged; only its `user_id` source changes.
- **Writes (create/edit/delete/append):** the author/owner is `resolve_identity(...)`. Authz coexists: **authenticated ‚Üí `author_id == your user id`**; **anonymous-sandbox ‚Üí A4's session-ownership** (unchanged). A guest (no identity) still can't write (existing 403).
- **The hole closes:** because the authenticated user always wins, **`preview_as` is ignored whenever a real user is logged in** ‚Äî no logged-in impersonation. `preview_as` works *only* for anonymous visitors and *only* under `SANDBOX_MODE`. Outside sandbox mode, an unauthenticated caller is a guest regardless of `preview_as`.

This is a behavior change worth stating: today (no auth) `preview_as` is honored unconditionally; after A5.1, `preview_as` is honored only for the anonymous sandbox demo. The existing suite (~100+ tests) passes `preview_as` and runs with `SANDBOX_MODE=False` by default ‚Äî so without action they'd all degrade to guest.

**Migration approach (chosen, to minimize churn):** those existing tests verify the *visibility + authz logic given an identity* ‚Äî which is unchanged; only the identity *source* moves. So a `conftest.py` autouse fixture flips `SANDBOX_MODE=True` for the existing API-test modules, keeping `preview_as` as the identity source there (their coverage is preserved verbatim). The **new** identity-resolution behavior (authenticated-bearer-wins; `preview_as` ignored while authed; guest when anonymous-and-not-sandbox) + the `/auth` endpoints get **dedicated new tests** that authenticate via a bearer-token helper. This avoids rewriting 100+ assertions while still proving the real auth path. The plan owns the exact fixture + the new tests.

## 4. Frontend

- `src/api/auth.ts`: `signup`/`login`/`logout`/`me`; stores the token in `localStorage["authToken"]`; a small accessor the existing `api/maps.ts` fetch wrappers use to attach `Authorization: Bearer <token>` when present. Clears the token on `401 from /auth/me` or logout.
- **Auth UI:** a login/signup form (a modal or a small screen). Logged in ‚Üí show `display_name`/email + a Logout control, and the app operates as the real user. Logged out ‚Üí the existing **`PreviewSwitcher`** persona demo remains (anonymous sandbox).
- On load: if a token exists, call `/auth/me` ‚Üí restore the user or clear a stale token.
- A small **"demo logins"** hint (e.g. "Try: friend@demo.example / demo-pass") so visitors can experience the authenticated app with seeded content.

## 5. Seed / demo

`maps/seed.py` gives each persona an `email` (`owner@demo.example`, `friend@demo.example`, ‚Ä¶) + a known demo `password` (via `make_password`), so the demo-login path works against the seeded memberships/groups/notes. The demo password is surfaced in the UI hint (and `DEPLOY.md`).

## 6. Security

PBKDF2 (Django default hasher); password ‚â• 8 (no truncating max); generic login error (no enumeration); 256-bit token stored as SHA-256 with a 14-day expiry, deleted on logout; bearer header ‚Üí **no CSRF**; prod HTTPS already enforced (A4). Token-in-`localStorage` XSS exposure is mitigated by the app's existing content escaping (A1.3-polish) + short token lifetime; documented as a known trade-off of bearer-in-SPA.

## 7. Testing

- **Backend:** signup (creates user + session + token; duplicate email ‚Üí 409; weak password ‚Üí 422); login (correct ‚Üí token; wrong password OR unknown email ‚Üí generic 401; blank-password user can't log in); logout (session deleted ‚Üí next call 401); `me` (valid ‚Üí user; expired/absent/garbage token ‚Üí 401); identity resolution (authenticated bearer wins; `preview_as` ignored while authenticated; `preview_as` honored only when anonymous **and** `SANDBOX_MODE`; guest otherwise); a note's author is the authenticated user; token stored hashed (raw token absent from the DB).
- **Frontend:** auth client (token store/attach/clear); login + signup forms (success + error states); logged-in vs demo UI; `/auth/me` restore on load.
- Full gates (backend pytest+ruff+format+mypy+makemigrations; frontend test+lint+tsc+build; e2e).

## 8. PR decomposition

- **A5.1a ‚Äî backend auth + identity wiring:** `User` fields + `AuthSession` + migration; the `/auth` endpoints; `authed_user` + `resolve_identity`; wire reads/writes through the resolver; the `preview_as`-vs-sandbox test migration; seed creds. Ships a working auth API.
- **A5.1b ‚Äî frontend auth:** the auth client + token wiring; login/signup UI + logout; `/auth/me` restore; the demo-login hint. Ships the usable login experience.

## Out of scope (later A5 sub-projects)

Login rate-limiting / lockout / brute-force protection; password reset + email verification; what tenant/role a **new** (non-seed) signup gets (membership flows ‚Äî new accounts initially see only public content); multi-tenant row-level security; "manage your sessions"/revoke-all; OAuth/social login. (CSRF is not applicable given the bearer-token transport.)
