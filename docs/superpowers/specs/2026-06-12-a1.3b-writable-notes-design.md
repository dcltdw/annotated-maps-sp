# A1.3b — Writable demo: own-note CRUD — Design

- **Status:** approved (brainstorming) — 2026-06-12
- **Relationship:** sub-slice of the [Annotated Maps design](2026-06-08-annotated-maps-design.md) (§5 domain, §6 visibility, §8 editable demo). Builds directly on **A1.3** (read map UI + live "Viewing as" switcher) and **A1.2** (models + create/soft-delete API). Read-only A1.3 stays intact; this adds the author loop.
- **Decisions locked in brainstorming:** full own-note CRUD (not just create); "Viewing as" persona *is* the author; whole-note `PUT` replace for edits; Audience picker exposes groups only; a small `groups` discovery endpoint; append/regions/collections/history deferred.

## 1. Goal & scope

Make the seeded Boston map **writable**: the selected persona can **create, edit, and soft-delete its own point-notes**, including each section's visibility rule + teaser, directly in the browser. Guests and non-authoring personas stay read-only.

**In scope:** create / edit / soft-delete own point-notes; the in-panel note editor with a per-section visibility control (Public / Audience / Reputation ≥ N / Private) + optional teaser; group picker for Audience; threshold for Reputation; optimistic-concurrency-guarded edits; a `groups` discovery endpoint; the backend author/guest authorization for writes; tests (backend authz suite, frontend unit, Playwright e2e).

**Out of scope (later slices):** append-a-child-note-to-others; region / whole-map / free-floating notes; collections / labels / layers; revision history; real authentication (the `preview_as` seam stays); rate/abuse limits (A5 owns them — this is their seam).

## 2. Identity model

No real auth yet, so authoring rides the existing **`preview_as` seam** (still `FIXME(A5)`):

- The currently selected **"Viewing as" persona is the author.** A note created while viewing as persona P is authored by P.
- A persona may **edit/delete only the notes it authored** (own-note). Viewing as a different persona, or as **Guest**, exposes **no write affordances** — fully read-only.
- A write request with **no `preview_as`** (guest) is rejected **403** at the API, independent of the UI hiding the controls.

## 3. The note editor (frontend)

A single presentational editor component, used for both create and edit, rendered in the note-panel slot of the A1.3 map screen.

**Placement & lifecycle.**
- **Create:** with a non-guest persona active, clicking an empty spot on the map drops a **draggable draft pin** and opens the editor (create mode). Dragging the pin updates the note's location.
- **Edit:** an **✎** affordance on one of your own notes reopens the same editor **pre-filled** (edit mode).
- **Delete:** a **⋯ → Delete** on your own note performs a **soft delete** after a confirm.
- **Cancel/✕** discards the draft/changes and returns to the read view.

**Editor structure.** Title field; an ordered list of **section editors**; "**+ Add section**"; **Save** / **Cancel**.

**Per-section control (the core UX).** Each section has **one visibility control** — four mutually-exclusive buttons (Public / Audience / Reputation ≥ N / Private) — that **live-reshapes** the card when clicked:
- **Public** → "visible to everyone"; **no teaser** (nothing is concealed).
- **Audience** → a **group multiselect** sourced from the tenant's groups (e.g. "Running club", "Close friends"); teaser available.
- **Reputation ≥ N** → a numeric **threshold** field; teaser available.
- **Private** → "only you — not other curators, not contributors"; teaser available.
- The card's accent + active button recolor to the rule's color (Public green / Audience teal / Reputation purple / Private gray — the A1.3 `ruleColors`).
- **Teaser** (for non-Public rules): an optional text shown locked to people who can't see the section; **default empty = nothing shown**.

**Read-only enforcement (UI).** Guests and personas that didn't author a note see **no** "+ new note", ✎, ⋯, or "+ Add section" — the panel is the A1.3 read-only view. (The backend enforces this independently; the UI is convenience, not the security boundary.)

## 4. Backend API (Django-Ninja, extends A1.2)

- `POST /maps/{map_id}/notes` — **exists** (create; `preview_as`=author). Reused.
- **`PUT /notes/{note_id}` — NEW.** Replace **title + point + sections** wholesale (the editor saves the whole note as one atomic form). Guarded by the **`version`** optimistic-concurrency field: the body carries the expected version; a mismatch returns **409**. **Author check:** only the note's author (current `preview_as` persona) may edit, else **403**. On success, bumps `version`, returns the updated note.
- `DELETE /notes/{note_id}` — **exists** (soft delete). Confirm/ensure it enforces **author-only** (403 otherwise).
- **`GET /maps/{map_id}/groups` — NEW (small).** Returns the tenant's groups (`id`, `name`) for the Audience picker — mirrors the A1.3 `viewers` discovery endpoint. Read-only.

**Validation** (Ninja schemas, extending the existing `SectionIn` / `NoteIn`, reusing the fail-closed rule mapping): title required; **≥ 1 section**; **content required (non-empty) for every section**; valid **rule params** (Audience needs ≥ 1 group id; Reputation needs a numeric threshold). Violations → **422** (no stored-DoS; consistent with A1.2's hardening).

**Authoring & limits.** Writes are attributed to `preview_as` (the seam, `FIXME(A5)`); guest writes → **403**. **No rate/abuse limits in this slice** — A5 owns them; the per-account/per-IP throttling and creation caps from the main design §8 attach at this same write path later.

## 5. Data flow

The editor assembles a `NoteIn` — `title`, `lng`/`lat` (from the pin), and `sections[]` each with `rule_type`, `rule_params` (group_ids / threshold), `content`, `teaser` — and:
- **Create:** `POST /maps/{id}/notes?preview_as=P` → on `201`, re-fetch notes (re-applies the visibility slice for P) → map markers + panel update.
- **Edit:** `PUT /notes/{id}?preview_as=P` with the loaded `version` → on success, re-fetch.
- **Delete:** `DELETE /notes/{id}?preview_as=P` → on `204`, re-fetch.

Re-fetching (rather than mutating client state) keeps the displayed slice authoritative — the same pattern A1.3 uses for the persona switch.

## 6. Errors & concurrency

- **Version conflict (409):** edit sends the loaded `version`; on 409 the editor surfaces **"This note changed elsewhere — reload to edit."** and does not corrupt local state.
- **Validation (422):** surfaced as inline messages on the offending field(s); submit is blocked client-side first (cheap guard) with the server as the real gate.
- **Guest/non-author write (403):** should never be reachable through the UI (controls hidden); the guard exists as defense-in-depth.

## 7. Testing

Docs + tests move with code (PR rigor). Three layers:

**Backend — extend the always-green authorization suite:**
- author-can-edit-own (PUT 200); non-author-cannot-edit (PUT 403); guest-cannot-write (POST/PUT/DELETE 403).
- version-conflict → 409; invalid input → 422 (missing title, zero sections, Audience without a group, Reputation without a threshold).
- soft-delete is author-only; deleted notes leave the listing.
- `GET /maps/{id}/groups` returns the tenant's groups.

**Frontend — Vitest component/unit (fast, no real maplibre):**
- the **interactive visibility control** — clicking Public/Audience/Reputation/Private reshapes config + teaser correctly.
- **validation messaging** — empty title / Audience-without-group / etc. → inline errors, blocked submit.
- **payload building** — a `NoteIn` is assembled correctly from editor state (pure-function test).
- create / edit / delete flows wired against a mocked API; guest & non-author hide the write controls.

**Playwright e2e — real maplibre in headless Chromium, with a *stateful* network stub.**
The stub shares a mutable `notes` array (test closure) and branches on HTTP method, so writes reflect in later reads — deterministic, no backend/DB/tiles:
- `**/api/v1/maps/*/notes` — GET returns current notes (by `preview_as`); POST parses `postDataJSON()`, assigns id + `version:1`, pushes, returns 201.
- `**/api/v1/notes/*` — PUT checks/bumps version & replaces (or forced 409 for the conflict case); DELETE removes, returns 204.
- `**/api/v1/maps/*/groups` → groups; `**/maps`, `**/viewers`, and the OpenFreeMap style stubbed as in A1.3-polish; marker click via dispatched native event (the polish workaround).

e2e cases (the six real browser flows):
1. **Create** — as owner: click empty map → editor → title → Section 1 Public + content → "+ Add section" → Section 2 Audience → pick "Running club" → content → Save → new marker appears, click shows both sections, and the POST payload is correct.
2. **Edit own** — seed an owner note → ✎ → editor pre-filled → change title + a rule → Save → PUT carried the loaded `version`; panel reflects the change.
3. **Version conflict** — force PUT → 409 → "changed elsewhere — reload" surfaced, no corruption.
4. **Delete own** — ⋯ → Delete → confirm → marker disappears, DELETE called.
5. **Guest read-only** — clicking the map opens no editor; existing notes show no ✎/⋯.
6. **Non-author read-only** — viewing as A, a note authored by B shows no ✎/⋯.

The new `e2e` CI job (added in A1.3-polish) runs these; it stays a PR check, not yet a required merge gate.

## 8. Seams & foundation touched

- **`preview_as`** remains the identity seam (`FIXME(A5)`), now also the *author* identity — replaced by real sessions in A5.
- The **write path** is the attach point for A5's per-account/per-IP throttling and creation caps (none built here).
- The visibility engine, A1.2 omission/null guarantees, and the read UI are **unchanged** — this slice only adds the write surface on top.
