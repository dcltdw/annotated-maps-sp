# A1.3c — Append-to-others (contributor child notes) — Design

- **Status:** approved (brainstorming) — 2026-06-17
- **Relationship:** sub-slice of the [Annotated Maps design](2026-06-08-annotated-maps-design.md) (§5 "Append"; §3 contributors append to a curator's note). Builds directly on **A1.3b** (own-note CRUD: the editor, the section visibility, the write path, the `preview_as` author seam) — reuses ~80% of it. The A1.3b read/write behavior is unchanged; this adds the append surface on top.
- **Decisions locked in brainstorming:** an append is a **full-note child** (own author + sections + per-section visibility), reusing the editor; **one level deep**; **panel-only** (no separate map marker); **no point**; **optional title** (the author name is the heading); rendered inline with a **configurable indent** and **author-name attribution** (avatars deferred, seam left); each append is **visibility-filtered independently**.

## 1. Goal & scope

Let a non-guest persona **append a child note to *any* note** (its own or another author's) — a first-class note (optional title + one-or-more sections, each with its own visibility rule + teaser) attached to a parent, **rendered inline** under it, **never mutating the parent**. The appender owns their append (edit/delete their own); the parent author is not modified and (in this slice) does not moderate others' appends.

**In scope:** the `Note.parent` relationship; append create/edit/delete; appends nested + visibility-filtered in the read API; the inline append rendering (configurable indent, author-name attribution, optional title) + the append affordance; tests (backend authz, frontend unit, Playwright e2e).

**Out of scope (later slices):** parent-author **moderation** of others' appends (A5); **appends-on-appends** (nesting > 1 level); **avatars** (the attribution is a seam — name-only now, expandable to name / avatar / name+avatar); region/whole-map notes (A2); real auth (the `preview_as` seam stays, `FIXME(A5)`); rate/abuse limits (A5).

## 2. Model & identity

- **`Note.parent`** — a self-referential FK: `ForeignKey("self", null=True, blank=True, on_delete=models.CASCADE, related_name="appends")`. An **append** is a `Note` with `parent` set; a **top-level note** has `parent IS NULL`. (Cascade delete: removing a parent removes its appends — acceptable for the demo; soft-delete of a parent hides it + its appends from the default-manager queries.)
- An append carries: `parent`, `author` (the appender), `map = parent.map`, **`title=""`** (optional), **`point=None`** (panel-only), and **sections** (≥1).
- **Identity (reuses the `preview_as` seam):** any **non-guest** persona may append to any note (guest → 403). The append is authored by `preview_as`. The appender may **edit/delete only their own** appends (the existing author check). One level only: appending to a note that already has a parent is rejected.
- **Visibility independence:** each append's sections run through the **same** `section_visibility` engine for the current viewer, keyed on the **append's** `author_id` as owner. So a contributor's **Private** append section is appender-only — the parent author does **not** see it; a guest sees only Public appends. This is the section-level permission model in a multi-author scenario.

## 3. Backend API (Django-Ninja, extends A1.3b)

- **`POST /notes/{parent_id}/appends`** — create an append. Body **`AppendIn` = `{ title: str = "", sections: list[SectionIn] }`** (no point; `title` optional — distinct from `NoteIn` which requires it; `sections` reuse `SectionIn` validation: ≥1 section, non-empty content, audience needs ≥1 target). Guards: guest (`preview_as is None`) → **403**; `get_object_or_404` on parent → **404**; parent already an append (`parent.parent_id is not None`) → **400** (one level). Creates the child `Note(parent=parent, author=preview_as, map=parent.map, title=payload.title, point=None)` + sections. Returns **201 `{id}`**.
- **`PUT /appends/{id}`** — edit your own append. Body **`AppendUpdateIn` = `{ title: str = "", sections, version }`**. Author check (`author == preview_as` → else 403) + version guard (mismatch → **409**) + transactional section replace — the same pattern as `PUT /notes/{id}`. *Dedicated endpoint because the note write schemas require title+point, which appends don't have; this keeps the note schemas strict rather than overloading them with conditional validation.*
- **Reused as-is for appends:** `DELETE /notes/{id}` (soft-delete, author-only) and `GET /notes/{id}/edit` (raw pre-fill — for an append it returns `title=""`/`lng=lat=null`/sections/version; the append editor uses the title + sections + version and ignores the null location).
- **Read — `list_notes`:** return only top-level notes (`the_map.notes.filter(parent__isnull=True)`), each `NoteOut` gaining **`appends: list[AppendOut]`**. **`AppendOut` = `{ id, author_id, author_name, title, sections: list[SectionOut] }`** (`author_name` = the appender's `display_name`). Each append's sections are visibility-filtered exactly like a note's (content nulled for teasers, `teaser_text` only for locked, fully-hidden sections dropped); an append with **no** visible/teaser sections is **omitted**. Prefetch appends + their authors/sections to avoid N+1.

## 4. Frontend (extends A1.3b)

- **types** (`api/types.ts`): `AppendOut { id, author_id, author_name, title, sections: SectionOut[] }`; `NoteOut.appends: AppendOut[]`; `AppendInput { title, sections }`; `AppendUpdateInput extends AppendInput { version }`.
- **api client** (`api/maps.ts`): `createAppend(parentId, AppendInput, previewAs)` → `POST /notes/{parentId}/appends`; `updateAppend(appendId, AppendUpdateInput, previewAs)` → `PUT /appends/{appendId}`; reuse `deleteNote` + `fetchNoteForEdit` for appends.
- **`NotePanel`:** below the parent's own sections, an **appends block** — a `↳ N appends` divider, then each append: an **author-name** heading + **optional title** + its visibility chips/sections; **✎ / ⋯** shown only when `canWrite && append.author_id === previewAs`; a **＋ Append** button when `canWrite`. The append indent is the CSS custom property **`--append-indent`** (default `14px`, set to `0` to flush left). The author display is a small **author-display seam** (name-only now; future name/avatar/name+avatar) so avatars don't touch the append layout later.
- **`NoteEditor`:** a new **append variant** (prop, e.g. `variant: "note" | "append"`) — hides the location, makes the **title optional** (no required-title validation), sections as usual. Emits `{ title, sections }` (create) or `{ title, sections, version }` (edit).
- **`MapScreen`:** the ＋Append affordance opens the editor in append-variant for the selected parent → on save `createAppend(parent.id, …, previewAs)` → reload notes. ✎ on your own append → `fetchNoteForEdit(append.id, previewAs)` → editor (append-variant) → `updateAppend`. ⋯ → `deleteNote(append.id, previewAs)` → reload. Reuses the existing guarded `loadNotes` + 409 handling.

## 5. Data flow & errors

Create/edit/delete an append → POST/PUT/DELETE as `preview_as` → on success **re-fetch notes** (the read API re-nests + re-filters appends for the active viewer) → panel re-renders. Errors reuse A1.3b's handling: version-conflict 409 → "changed elsewhere" message; validation 422 (inline); guest/non-author write → 403 (UI hides the affordances; backend is the real gate). Appending to an append → 400 (the UI only surfaces ＋Append on top-level notes, so this is a defensive guard).

## 6. Testing

- **Backend** (extend the always-green authorization suite): append create (non-guest 201; guest 403); **can't append to an append** (400); edit/delete **own** append (200/204); **non-author** edit/delete → 403; **append sections visibility-filtered independently** — a Private append section is hidden from the parent author and from other viewers but visible to its own author (the key authz test); fully-hidden appends omitted; `list_notes` nests appends under parents and only returns top-level notes; `AppendOut.author_name` resolves.
- **Frontend unit (Vitest):** NotePanel append rendering (own → ✎/⋯, others → read-only; author-name + optional-title; ＋Append only when canWrite); NoteEditor append variant (optional title, no location field, builds the append payload); api client URLs (`createAppend`/`updateAppend`); MapScreen append/edit/delete wiring against a mocked API.
- **Playwright e2e** (extend the stateful author-loop stub with appends): append a note to a parent as a contributor → it appears inline under the parent; edit + delete your own append; **a Private append by persona A is not visible to persona B** (visibility independence verified in a real browser).

## 7. Seams & foundation touched

- `preview_as` remains the identity/author seam (`FIXME(A5)`), now also the append author.
- The **author-display** in the append attribution is a seam for later avatars (name → name/avatar/name+avatar) without changing the append layout.
- The visibility engine, A1.2/A1.3/A1.3b read & write guarantees, and the map-marker layer are **unchanged** — appends are panel-only and reuse the existing section-visibility path.
