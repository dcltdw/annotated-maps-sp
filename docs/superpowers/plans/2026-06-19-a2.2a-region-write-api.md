# A2.2a — Region Notes: Write API — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let the API create and edit region notes — accept a polygon/line `shape` (or a point) on write, build the right geometry, and enforce "exactly one anchor" at both the serializer and the database.

**Architecture:** `NoteIn` gains an optional `shape` and makes `lng`/`lat` optional, with a validator requiring exactly one anchor. A `_anchor_fields(payload)` helper turns the anchor into a `{point, area, path}` dict (one set, two `None`) used by both `create_note` and `update_note` (so editing can also *convert* a note's anchor type). `note_for_edit` returns the `shape` so the editor (A2.2b) can load a region. A DB `CheckConstraint` enforces the invariant as defense-in-depth (appends, which carry no anchor, are exempt).

**Tech Stack:** Django 5.1 + GeoDjango (Point/Polygon/LineString), Django-Ninja (pydantic validators), pytest.

**Spec:** `docs/superpowers/specs/2026-06-19-a2-region-notes-design.md` (§5). This is **PR A2.2a** (backend write); **A2.2b** adds the `ShapeDrawer` draw abstraction + the polygon authoring UI; A2.3/A2.4 add line/circle. The frontend still only sends point notes until A2.2b, so this PR is backend-only and backward-compatible.

**Conventions:** backend from `backend/`, `uv run …`; DB up via `docker compose up -d db` + wait for `pg_isready`. Run the FULL backend gate after each task: `uv run pytest && uv run ruff check . && uv run ruff format --check . && uv run mypy . && uv run python manage.py makemigrations --check --dry-run`. The `boston` fixture (in `test_notes_api.py`) returns `{"map","owner"}` with a point note titled "Beacon Hill"; `_note_with_sections(boston)` builds an editable point note (used by edit tests).

```python
class ShapeIn(Schema):
    kind: str  # "polygon" | "line"
    coordinates: list[tuple[float, float]]  # [lng, lat] pairs

    @field_validator("kind")
    @classmethod
    def _kind_valid(cls, v: str) -> str:
        if v not in ("polygon", "line"):
            raise ValueError(f"invalid shape kind {v!r}")
        return v

    @model_validator(mode="after")
    def _enough_coordinates(self) -> "ShapeIn":
        if self.kind == "polygon" and len(self.coordinates) < 3:
            raise ValueError("a polygon needs at least 3 coordinates")
        if self.kind == "line" and len(self.coordinates) < 2:
            raise ValueError("a line needs at least 2 coordinates")
        return self
```

- [ ] **Step 4: Update `NoteIn`** — make `lng`/`lat` optional, add `shape`, and add the exactly-one-anchor validator. Keep the existing `title` + `sections` validators:
```python
class NoteIn(Schema):
    title: str = ""
    lng: float | None = None
    lat: float | None = None
    shape: ShapeIn | None = None
    sections: list[SectionIn] = []

    @field_validator("title")
    @classmethod
    def _title_nonempty(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("title is required")
        return v

    @field_validator("sections")
    @classmethod
    def _at_least_one_section(cls, v: list[SectionIn]) -> list[SectionIn]:
        if not v:
            raise ValueError("a note needs at least one section")
        return v

    @model_validator(mode="after")
    def _exactly_one_anchor(self) -> "NoteIn":
        has_point = self.lng is not None and self.lat is not None
        has_shape = self.shape is not None
        if has_point == has_shape:  # both, or neither
            raise ValueError("provide exactly one anchor: lng+lat OR shape")
        return self
```
(`NoteUpdateIn(NoteIn)` inherits all of this automatically — no change needed there.)

```python
def _anchor_fields(payload: NoteIn) -> dict:
    """Turn a NoteIn's anchor into a {point, area, path} dict (one set, two None).
    Raises HttpError(422) for invalid geometry. The exactly-one rule is already
    enforced by NoteIn's validator, so exactly one branch applies."""
    if payload.shape is not None:
        if payload.shape.kind == "polygon":
            ring = [(x, y) for x, y in payload.shape.coordinates]
            if ring[0] != ring[-1]:
                ring.append(ring[0])  # GEOS needs a closed ring
            poly = Polygon(ring)
            if not poly.valid:
                raise HttpError(422, "Invalid polygon (edges cross?).")
            return {"point": None, "area": poly, "path": None}
        return {"point": None, "area": None, "path": LineString(payload.shape.coordinates)}
    return {"point": Point(payload.lng, payload.lat), "area": None, "path": None}
```

## Definition of Done (A2.2a)

- [ ] The write API accepts a `shape` (polygon/line) or a point; `NoteIn` enforces exactly one anchor (422 on both/neither); invalid geometry (self-intersecting polygon) → 422.
- [ ] `create_note` + `update_note` build the right geometry; editing can **convert** a note's anchor type (point↔polygon↔line), clearing the old geometry.
- [ ] `note_for_edit` returns the `shape` (for A2.2b's editor).
- [ ] A DB `CheckConstraint` enforces the one-anchor invariant for top-level notes (appends exempt); `Note.Meta` subclasses `TenantScopedModel.Meta` (soft-delete manager preserved).
- [ ] Backend gate green (pytest + ruff + format + mypy + makemigrations). Existing point-note create/edit behavior unchanged (backward compatible). No frontend change.
