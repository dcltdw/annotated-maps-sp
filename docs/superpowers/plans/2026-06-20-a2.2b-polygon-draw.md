# A2.2b — Region Notes: Draw Abstraction + Polygon Authoring — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let a user *draw a polygon* on the map and save it as a region note — via a swappable `ShapeDrawer` abstraction (terra-draw behind a port) so the drawing library can be replaced without touching the app.

**Architecture:** A `frontend/src/lib/draw/` port defines `ShapeDrawer` + our own `DrawShape` types; a `TerraDrawAdapter` is the *only* file that imports terra-draw; a `FakeShapeDrawer` lets every other test run without terra-draw/WebGL. MapScreen gains a "Draw area" entry point that runs the drawer, then opens the existing `NoteEditor` carrying the drawn `shape`, and saves via `createNote`/`updateNote` (which already accept `shape` server-side from A2.2a). A factory + a non-prod window override let the e2e inject the fake.

**Tech Stack:** React + TS + Vite; maplibre-gl 5; **terra-draw 1.x** + **terra-draw-maplibre-gl-adapter 1.x** (the unscoped adapter package — confirmed on npm).

**Spec:** `docs/superpowers/specs/2026-06-19-a2-region-notes-design.md` (§2 draw abstraction, §4 edit scope, §6 build sequence). This is **PR A2.2b**; A2.3 adds line authoring, A2.4 circle (both reuse this abstraction). Backend write support already shipped in A2.2a (`NoteIn.shape`, `note_for_edit` returns `shape`).

**Conventions:** frontend from `frontend/`. Gates: `npm run test -- --run`, `npm run lint` (`--max-warnings 0`), `npx tsc -b`, `npm run build`, `npm run e2e`. Read each file before editing — match real structure. Current create flow: click map → `handleMapClick(lng,lat)` sets `draft` + `mode="create"` → `NoteEditor` (lng/lat from draft) → `handleSave` → `createNote(map.id, {title,lng,lat,sections}, previewAs)`. `Mode = "view"|"create"|"edit"|"append"|"edit-append"`.

- [ ] **Step 1: Create the port** `frontend/src/lib/draw/types.ts`:
```ts
import type { Map as MaplibreMap } from "maplibre-gl";

export type DrawMode = "polygon" | "line" | "circle";

/** Our geometry type — independent of any draw library. `circle` mode emits a `polygon`. */
export type DrawShape =
  | { kind: "polygon"; coordinates: [number, number][] } // outer ring [lng,lat]
  | { kind: "line"; coordinates: [number, number][] };

/** A swappable map-drawing port. Only adapters import a concrete draw library. */
export interface ShapeDrawer {
  mount(map: MaplibreMap): void;
  startDraw(mode: DrawMode, onComplete: (shape: DrawShape) => void): void;
  editShape(shape: DrawShape, onChange: (shape: DrawShape) => void): void;
  cancel(): void;
  destroy(): void;
}
```

- [ ] **Step 4: Create** `frontend/src/lib/draw/FakeShapeDrawer.ts`:
```ts
import type { DrawMode, DrawShape, ShapeDrawer } from "./types";

/** Test double: drives onComplete/onChange synchronously via emit(); no map/WebGL needed. */
export class FakeShapeDrawer implements ShapeDrawer {
  lastMode: DrawMode | null = null;
  private onComplete: ((s: DrawShape) => void) | null = null;

  mount(): void {}
  startDraw(mode: DrawMode, onComplete: (s: DrawShape) => void): void {
    this.lastMode = mode;
    this.onComplete = onComplete;
  }
  editShape(_shape: DrawShape, _onChange: (s: DrawShape) => void): void {}
  cancel(): void {
    this.onComplete = null;
  }
  destroy(): void {
    this.onComplete = null;
  }
  /** Test helper — simulate the user finishing a shape. */
  emit(shape: DrawShape): void {
    this.onComplete?.(shape);
  }
}
```

- [ ] **Step 5: Create the factory** `frontend/src/lib/draw/index.ts`. It returns the real adapter in production, but honors a non-production window override so the e2e can inject a fake:
```ts
import type { ShapeDrawer } from "./types";

export type { DrawMode, DrawShape, ShapeDrawer } from "./types";
export { FakeShapeDrawer } from "./FakeShapeDrawer";

declare global {
  interface Window {
    __shapeDrawerOverride?: ShapeDrawer;
  }
}

/** Build the app's ShapeDrawer. A non-production window override (set by e2e) wins,
 *  so tests can inject a FakeShapeDrawer without bundling/initialising terra-draw. */
export async function createShapeDrawer(): Promise<ShapeDrawer> {
  if (import.meta.env.MODE !== "production" && window.__shapeDrawerOverride) {
    return window.__shapeDrawerOverride;
  }
  const { TerraDrawAdapter } = await import("./TerraDrawAdapter");
  return new TerraDrawAdapter();
}
```
(The dynamic `import("./TerraDrawAdapter")` keeps terra-draw out of the main bundle — it's lazy-loaded like maplibre. The adapter file is created in Task 2; until then `createShapeDrawer` won't be called by any shipping code, so this compiles once Task 2 lands. To keep THIS task's `tsc`/build green without the adapter yet, temporarily point the import at a stub OR implement Task 2 before building — simplest: in this task, make the dynamic import resilient by wrapping the adapter import in Task 2. For now, create a placeholder `TerraDrawAdapter.ts` exporting a class that throws `new Error("not implemented")` from its methods so the module resolves; Task 2 replaces it.)

## Task 2: `TerraDrawAdapter` — integration spike (de-risk gate)

**Files:**

This task is the **de-risk gate** the spec calls for. terra-draw's exact v1 API must be read from its own type definitions (do NOT guess) — the adapter is the only place it's used, so if terra-draw + maplibre 5 don't cooperate, only this file changes (or we swap to a hand-rolled adapter behind the same port).

- [ ] **Step 1: Install** the library + maplibre adapter:
```bash

```

- [ ] **Step 2: Read terra-draw's v1 API** from `node_modules/terra-draw/dist/terra-draw.d.ts` and `node_modules/terra-draw-maplibre-gl-adapter/dist/*.d.ts`. Identify: the `TerraDraw` constructor (it takes `{ adapter, modes }`), the maplibre adapter class (constructed with `{ map }` or `{ lib, map }`), `TerraDrawPolygonMode`, `TerraDrawLineStringMode`, `TerraDrawCircleMode`, and how to start a mode (`draw.setMode(...)`), read finished features (the `"finish"` event / `draw.getSnapshot()` → GeoJSON FeatureCollection), and `draw.start()/stop()`. Note the actual names you find.

- [ ] **Step 3: Implement** `frontend/src/lib/draw/TerraDrawAdapter.ts` to satisfy the `ShapeDrawer` port using terra-draw. Contract (adapt the terra-draw specifics to what Step 2 found):
  - `mount(map)`: construct `new TerraDraw({ adapter: new TerraDrawMapLibreGLAdapter({ map, lib: maplibregl }), modes: [polygonMode, lineMode, circleMode] })` and `draw.start()`. Store the instance.
  - `startDraw(mode, onComplete)`: map our `DrawMode` → the terra-draw mode name (`"polygon"`/`"linestring"`/`"circle"`); `draw.setMode(name)`; subscribe to the finish event; on finish, read the just-completed feature's GeoJSON geometry and convert to our `DrawShape`:
    - polygon (and **circle** — terra-draw's circle mode emits a `Polygon`): `{ kind: "polygon", coordinates: feature.geometry.coordinates[0] }` (outer ring, [lng,lat]).
    - linestring: `{ kind: "line", coordinates: feature.geometry.coordinates }`.
    - call `onComplete(shape)`, then `draw.setMode("static")` (or the library's idle mode) and clear the drawn feature so it doesn't linger (display is handled by MapView's `regions` layers after save).
  - `editShape(shape, onChange)`: load the shape as a feature in select/edit mode and call `onChange` on vertex edits — IF terra-draw's select mode integrates cleanly. If it's fiddly, leave `editShape` to throw/no-op and the UI will use redraw-to-change (the spec allows this fallback); report which you did.
  - `cancel()`: `draw.setMode("static")` + clear in-progress feature.
  - `destroy()`: `draw.stop()`.
  - **Confirm circle → polygon:** terra-draw's `TerraDrawCircleMode` produces a `Polygon` (a many-sided ring) — verify in Step 2's types / a quick check; this is what A2.4 relies on. Report the finding.

- [ ] **Step 4: Verify the integration (manual spike — there is no robust headless unit test for a WebGL draw library):**
  - `npm run dev`, open the app, and confirm (this is the gate): entering polygon-draw mode lets you click out vertices and finish a polygon, and the adapter's `onComplete` fires with a `{kind:"polygon", coordinates}` ring. (You can wire a temporary dev-only button or test via Task 4's UI once it exists — but at minimum verify the adapter constructs + `start()`s against the real map without errors.)
  - Add a lightweight smoke check: a vitest test that `new TerraDrawAdapter()` constructs and `mount` doesn't throw against a minimal maplibre mock (or, if the adapter requires a real GL map, skip the unit test and rely on the manual check + the e2e in Task 5). Report what you used.
  - **If terra-draw + maplibre 5 is fundamentally incompatible** (adapter errors on mount, modes don't fire): STOP and report BLOCKED — we either pin a compatible version or hand-roll the adapter behind the same `ShapeDrawer` port (the abstraction makes this a localized change).
