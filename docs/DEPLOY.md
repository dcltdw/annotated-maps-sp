# Deploy Runbook — Annotated Maps Public Sandbox

This runbook covers a first-time deploy of the Annotated Maps public sandbox to
**Render** (web service + static site + cron) backed by **Neon Postgres/PostGIS**.
The deploy is manual; you need accounts on both platforms.

---

## 1. Neon — Create the database

### Via the Neon dashboard (recommended)

1. Log in at <https://console.neon.tech> → **New project**.
2. Choose a name (e.g. `annotated-maps`), region nearest your users, and the
   latest Postgres version.
3. Open the **SQL Editor** for the default branch and run:
   ```sql
   CREATE EXTENSION IF NOT EXISTS postgis;
   ```
4. Go to **Connection Details** → copy the **pooled connection string**
   (looks like `postgres://user:pass@ep-…-pooler.us-east-2.aws.neon.tech/neondb?sslmode=require`).
   This is your `DATABASE_URL`.

### Optional scripted path (requires `neonctl auth`)

```bash
neonctl projects create --name annotated-maps --region-id aws-us-east-2
# note the project-id in the output, e.g. "quiet-snow-12345678"
neonctl connection-string quiet-snow-12345678 --pooled
```

Run the PostGIS extension via the dashboard or with `psql "$DATABASE_URL" -c "CREATE EXTENSION IF NOT EXISTS postgis;"`.

---

## 2. Render — New Blueprint deploy

### Via the Render dashboard (recommended)

1. Log in at <https://dashboard.render.com> → **New +** → **Blueprint**.
2. Connect your GitHub account if not already connected; select the
   `annotated-maps-sp` repo.
<!-- fact: tier=pr cmd="yq '.services | length' render.yaml" expect="3" prose="three" -->
3. Render detects `render.yaml` at the repo root and shows three services:
   - **`annotated-maps-api`** — Docker web service (Django/Ninja API)
   - **`annotated-maps-web`** — Static web service (React/Vite frontend)
   - **`annotated-maps-reaper`** — Docker cron job (daily ephemeral note reaper)
4. Click **Apply** / **Create Blueprint**.

Render will queue initial builds but the services will fail until you set the
`sync: false` env vars in the next step.

### Optional scripted path (requires `RENDER_API_KEY`)

After the Blueprint is applied and service IDs are known, you can set env vars
via the Render API:

```bash
# Replace SERVICE_ID with the actual service id from the dashboard URL.
curl -s -X PATCH \
  "https://api.render.com/v1/services/SERVICE_ID/env-vars" \
  -H "Authorization: Bearer $RENDER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '[{"key":"DATABASE_URL","value":"<your-neon-url>"}]'
```

## 3. Env vars to set (the `sync: false` ones)

These must be set manually — they are not committed to the repo.

### API service — `annotated-maps-api`

| Variable | Value |
|---|---|
| `DATABASE_URL` | Neon pooled connection string from step 1 |
| `DJANGO_ALLOWED_HOSTS` | The API hostname, e.g. `annotated-maps-api.onrender.com` |
| `CORS_ALLOWED_ORIGINS` | The web service URL, e.g. `https://annotated-maps-web.onrender.com` |
| `MOD_TOKEN` | *(optional)* long random string; enables the `/moderate` tooling — moderation stays off if unset |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | *(optional)* OpenTelemetry collector endpoint for the Grafana Cloud export (Milestone 2) — telemetry export stays off if unset |
| `OTEL_EXPORTER_OTLP_HEADERS` | *(optional, secret)* carries the Grafana Cloud auth token |

The following are already set by `render.yaml` and need no action:
- `DJANGO_SECRET_KEY` — auto-generated by Render on first deploy
- `DJANGO_DEBUG` — `"false"`
- `SANDBOX_MODE` — `"true"`

The health endpoint's `git_sha` field reads Render's built-in `RENDER_GIT_COMMIT`
automatically — no env var to set.

### Reaper cron — `annotated-maps-reaper`

| Variable | Value |
|---|---|
| `DATABASE_URL` | Same Neon connection string as the API |

`DJANGO_SECRET_KEY` needs **no action** — `render.yaml` sets it to `generateValue: true`, so Render generates an independent key. It need not match the API's: the reaper only hard-deletes old rows and never verifies signed cookies or tokens.

`SANDBOX_MODE=true` is already set in `render.yaml`.

### Sandbox abuse caps

- Authenticated users (including the seeded demo-persona logins) are not subject to the
  per-session note/append caps, but the per-IP-hourly and global-row caps still apply to
  everyone in `SANDBOX_MODE` — so the public demo-persona password cannot be used to spam.

### Static web service — `annotated-maps-web`

| Variable | Value |
|---|---|
| `VITE_API_BASE` | Full API URL including path prefix, e.g. `https://annotated-maps-api.onrender.com/api/v1` |

`VITE_SANDBOX=true` is already set in `render.yaml`.

After setting the env vars, trigger a **Manual Deploy** for each service (or
wait for the next auto-deploy from a git push).

## 4. Migrations + seed data — automatic

The API service runs migrations and rebuilds the demo seed before every deploy.
Render does not run `preDeployCommand` through a shell (so `&&` can't chain), so
<!-- fact: tier=pr cmd="grep -c 'preDeployCommand: sh predeploy.sh' render.yaml" expect="1" prose="predeploy" -->
`render.yaml` sets `preDeployCommand: sh predeploy.sh`, and `backend/predeploy.sh` runs:

```
uv run python manage.py migrate
uv run python manage.py seed_demo --refresh
```

Render runs this automatically **before every deploy**, so migrations are applied
and the demo seed is rebuilt with no manual intervention. `--refresh` deletes the
existing seed notes (`is_seed=True`, cascading their sections/appends) and recreates
them from the current code, so changes to `maps/seed.py` take effect on the next
deploy. User-created notes (`is_seed=False`) and the personas/map/groups are left
intact, so visitor content and logins survive a re-seed.

Seed notes are read-only in sandbox mode. To re-apply seed changes without a code
push, trigger a Manual Deploy, or run `uv run python manage.py seed_demo --refresh`
from the `annotated-maps-api` **Shell**.

## 5. Smoke test

Run the automated post-deploy verifier from your local machine:

```bash
python3 scripts/smoke_test.py https://annotated-maps-api.onrender.com/api/v1
```

The script exercises (in order):
1. `GET /health` — asserts `status == "ok"`, prints `version` + `git_sha`.
2. `GET /maps` — asserts ≥1 map, captures the first map id.
3. `GET /maps/{id}/viewers` — asserts ≥1 viewer, captures the first as persona.
4. `GET /maps/{id}/notes?preview_as={persona}` — asserts the Castle Island seed
   note has `editable: false`.
5. `POST /maps/{id}/notes?preview_as={persona}` — creates a smoke-test note,
   asserts HTTP 201.
6. `GET /maps/{id}/notes?preview_as={persona}` again — asserts the new note has
   `editable: true` (session-owned).
7. `DELETE /notes/{new_id}?preview_as={persona}` — deletes the note it created so
   the smoke test leaves no pin behind in the sandbox; asserts HTTP 204.

On success all steps print `OK  step N: …` and the script exits 0.
Any failure prints `FAIL step N: …` to stderr and exits 1.

### Manual browser pass

After the smoke test passes:

1. Open `https://annotated-maps-web.onrender.com` → the Boston/Castle Island
   note should render on the map.
2. Use the "Viewing as" selector — locked sections should appear/disappear
   depending on the persona's visibility rules.
3. Create a note as a persona → it appears on the map and shows `editable` in
   the note panel for you; open an **incognito window** and verify the same
   note is **not** editable there (no session cookie).
4. The sandbox banner should be visible in the UI (`VITE_SANDBOX=true`).

## 6. Cron check — reaper schedule

Confirm `annotated-maps-reaper` is registered and active in the Render
dashboard under **Cron Jobs**. The schedule from `render.yaml` is:

```
17 4 * * *    (daily at 04:17 UTC)
```

The cron runs `uv run python manage.py reap_ephemeral`, which deletes ephemeral
notes older than 7 days.

You can trigger a one-off run manually via Render dashboard → Cron Job →
**Run Now** to verify it connects to the database without errors.

---

## 7. Caveats

- **Render free-tier spin-down**: Web services on the free tier spin down after
  15 minutes of inactivity. The first request after idle incurs a cold-start
  delay of 30–60 seconds. Upgrading to a paid instance type eliminates this.

- **Cron availability by plan**: `annotated-maps-reaper` requires a plan that
  includes Render Cron Jobs. If the cron is not available on your chosen plan,
  run reaping via an external scheduler (e.g. GitHub Actions scheduled workflow
  or a cron on any always-on machine):
  ```bash
  # From a machine with DATABASE_URL and DJANGO_SECRET_KEY set:
  uv run python manage.py reap_ephemeral
  ```
  Alternatively, run it manually from the Render Shell on the API service.

- **PostGIS extension**: Must be enabled in Neon before migrations run.
  If `migrate` fails with `type "geography" does not exist`, run
  `CREATE EXTENSION IF NOT EXISTS postgis;` in the Neon SQL Editor and redeploy.

- **Seeding idempotency**: `seed_demo` uses `get_or_create` on the note title,
  so re-running it is safe but redundant. The demo map and viewer are also
  created idempotently.

- **CORS**: `CORS_ALLOWED_ORIGINS` must exactly match the web service origin
  (including scheme, no trailing slash). A mismatch shows up as CORS errors in
  the browser console when the frontend calls the API.
